## 1. Обзор текущего опыта участников

### 1.1 Antigravity vs Claude Code — сравнение скоростей
- Участник протестировал **одинаковый промпт** одновременно в Antigravity и Claude Code (с 5 агентами)
- **Antigravity оказался быстрее**, чем Claude Code с мультиагентной системой
- Оба используют модель **Opus**, но подходят к задаче по-разному
- **Нет шаблона** — каждый раз агенты решают задачу иначе

### 1.2 Контексты и ограничения
- Несмотря на заявленный миллион токенов, контекст **рано или поздно переполняется**
- При зависании: открыть **новый чат** и попросить изучить файлы проекта
- Antigravity умеет **подхватывать контекст из предыдущего чата**
- Альтернатива: скопировать весь чат и вставить в новый (метод из Claude)

### 1.3 Мостик Claude Code → Telegram
- Можно **прокинуть мостик** из терминала Claude Code в Telegram
- Не нужны API или дополнительные токены
- Можно **работать и подтверждать задачи** прямо из Telegram (Run, Integrity)
- Оставить компьютер включённым — агент работает автономно

### 1.4 Скиллы — результаты проверки
- Скиллы, созданные через **Skill Creator**, оказались **лучше** скачанных из маркетплейса
- После небольших доработок — скиллы стали практически **идеальными**
- Важно правильно прописывать **description** скилла — от этого зависит автоматическое подключение

## 2. Три способа создания дизайна

### 2.1 Способ 1: Описание словами
- Открыть Antigravity → описать, каким должен быть дизайн
- Результат: **стандартный шаблонный сайт**
- Подходит для начальной версии, но требует доработки

### 2.2 Способ 2: Через исследование (Deep Research)
- Использовать **Gemini, GPT, Perplexity** для глубокого исследования
- Попросить исследовать **сайты конкурентов**, оценить дизайн
- ИИ соберёт: тексты, гайдлайны, шрифты, структуру
- Можно добавить информацию о своём продукте для полной картины
- Также можно просто дать **ссылку на понравившийся сайт** — ИИ сделает похожий

### 2.3 Способ 3: На основе существующего кода (рекомендуемый)
- Уже есть продукт → уже есть функционал и какой-то интерфейс
- Попросить ИИ **улучшить дизайн на основе имеющегося кода**
- Самый практичный подход для продуктов в разработке

## 3. Работа с ветками GitHub (Branches)

### 3.1 Зачем нужны ветки
- **Безопасная** работа в команде над одним проектом
- Каждый разработчик работает в **своей ветке**
- Основная ветка **Main** остаётся нетронутой
- После проверки — изменения **мерджатся** (Merge) в Main

### 3.2 Схема работы

```
Main (основная ветка)
  ↓
  ├── Разработчик 1: доработка бэкенда
  ├── Разработчик 2: ветка "сайт" (фронтенд)
  └── ...
  
После проверки → Merge → изменения попадают в Main
```

### 3.3 Практические шаги
1. **Скачать репозиторий:** дать Antigravity ссылку — «Разверни репозиторий GitHub»
2. **Создать ветку:** «Создай отдельную ветку, назови её сайт»
3. **Работать в ветке:** все изменения идут только в ветку
4. **Отправить код:** «Отправь код в ветку сайт»
5. **Главный разработчик проверяет** и делает Merge

### 3.4 Советы при развёртывании
- Указывать **конкретную папку** при развёртывании: «Разверни в файловой ветке»
- Или заранее **создать папку** с именем проекта
- По умолчанию репозиторий разворачивается в **корневую папку** Antigravity

## 4. Процесс создания дизайна — пошаговый алгоритм

### 4.1 Шаг 1: Изучение проекта
```
Изучи проект, весь функционал и расскажи,
что может этот сервис и для какой аудитории.
```
- ИИ анализирует весь проект и создаёт **контекст для дальнейшей работы**

### 4.2 Шаг 2: Проработка структуры
```
Буду разрабатывать дизайн главной страницы.
Сейчас на главной странице — только логин и пароль.
Пока не пиши код, давай проработаем структуру.
```

Типичная структура лендинга:
1. **Хедер** (шапка)
2. **Главный экран** (hero-секция)
3. **Преимущества**
4. **Основные фичи** (описание продукта)
5. **Демонстрация** (скриншоты, видео приложения)
6. **Тарифы** (если есть)
7. **Отзывы** (если есть)
8. **Футер**

### 4.3 Шаг 3: Текст по методологии Jobs to be Done
```
Я хочу, чтобы текст на главной странице был написан
по методологии Jobs to be Done.
```
- **JTBD** — ориентация на решение проблем пользователей
- Текст пишется от задач, которые пользователь хочет решить
- Есть расширенная версия — **Advanced JTBD (АДЖДБД)**

### 4.4 Шаг 4: Референсы
- Собрать скриншоты / ссылки на понравившиеся сайты
- Отправить ИИ: «Изучи сайт-референс и сделай лендинг с таким же дизайном»

### 4.5 Шаг 5: Генерация с правилами UX/UI
```
Сделай главную страницу. Сделай иерархию акцентов.
Соблюдай все правила UX и UI.
```

## 5. Ключевые принципы дизайна

### 5.1 Иерархия акцентов
- **Где подсветить — там и внимание.** Всё.
- Жирный текст = фокус внимания
- Не нужно запоминать сложные теории (F-паттерн, боли и т.д.)
- Достаточно дать промпт: **«Соблюдай иерархию акцентов»**

### 5.2 UX vs UI
| Аббревиатура | Расшифровка | Суть |
|---|---|---|
| **UI** | User Interface | Красота: кнопки, цвета, анимации |
| **UX** | User Experience | Удобство: куда нажимает, что видит |

- **UI** — «хаос и красота»: анимации, градиенты, элементы
- **UX** — логика пользовательского пути: кнопки на своих местах, интуитивно понятно

> 💡 Всегда прописывайте в промптах: **«Соблюдай UX и UI»**

### 5.3 Цитата Олега Янковского
> *«Убеждает логика, а не красота»*

## 7. Перенос дизайна из Google AI Studio в проект

### 7.1 Способ 1: Скачать и закинуть
1. В Google AI Studio нажать **Code → Скачать**
2. Разархивировать
3. Закинуть папку **в рабочую директорию** Antigravity
4. Сказать: «Вот я разработал лендинг, замени текущий на этот и запусти»

1. В Google AI Studio → **Publish → Интеграции → GitHub**
2. Авторизоваться → создать новый репозиторий
3. **Закоммитить** (обновить версию)
4. В Antigravity: «Разверни этот репозиторий»

## 8. Картинки и качество изображений

- Сгенерированные ИИ скриншоты/изображения могут быть **низкого качества**
- **Оптимизация делается в конце**, на этапе полировки
- Решения:
  - Подключить **Imagen 2** (Nano Banana 2) — генерирует качественные картинки
  - Найти фото через **парсинг** других сайтов
  - Использовать **generate_image** для создания нужных изображений

## 9. Бизнес-цикл продукта

### 9.1 Схема полного цикла

```
Исследование → Гипотеза/Идея → Разработка → Маркетинг → Продажи → Обратная связь
                                                                        ↓
                                                               → Исследование (цикл)
```

### 9.2 Три типа исследований
1. **Качественные** — личное общение с аудиторией (нельзя делегировать!)
2. **Количественные** — анкеты с 3–4 вопросами, обратная связь
3. **Онлайн-разведка (OSINT)** — добыча информации из открытых источников

### 9.3 Статистика гипотез
- **1 из 10** гипотез выстреливает (стандартная статистика)
- **3 из 100** — на репрезентативной выборке
- Раньше стоимость одной гипотезы: **~1 млн ₽**
- Сейчас (благодаря вайбкодингу): **~300-500 тыс. ₽** (включая маркетинг и трафик)
- **Отказ от нерабочей идеи — тоже экономия**

### 9.4 OSINT-разведка
- Существуют репозитории на GitHub с инструментами онлайн-разведки
- Отслеживание: движение транспорта, тренды, биржевые сигналы
- Отличие от Deep Research: **глубже, конфиденциальнее**, но из открытых источников
